# バッチレビュー・公開ワークフロー

## 概要

生成された記事をバッチとしてまとめ、Slack通知 → 人間の承認 → 公開までを管理する。

## 実行タイミング

`generate-articles.md` 完了後に実行

## 手順

### Step 1: GitHub PR 作成

生成された全記事をまとめて1つの GitHub PR として作成する。

```bash
# ブランチ作成
git checkout -b articles/YYYY-MM-DD

# 記事ファイルを追加
git add src/content/articles/YYYY-MM-DD-*.md

# コミット
git commit -m "Add daily articles for YYYY-MM-DD"

# プッシュ & PR作成
git push -u origin articles/YYYY-MM-DD
gh pr create --title "📰 Daily Articles: YYYY-MM-DD" --body "..."
```

### Step 2: Slack バッチ通知

以下のフォーマットで Slack チャンネルに通知する:

**投稿先**: TODO: チャンネルを指定

```
📰 *デイリーニュース候補（YYYY/MM/DD）* — {N}件

━━━━━━━━━━━━━━━━━
{各記事について}
{番号}{バーティカル絵文字} {バーティカル名}
*「{記事タイトル}」*
📊 品質スコア: {score}/100 | 🏷 {quality_tier}
📝 プレビュー: {GitHub PR内のファイルリンク}

━━━━━━━━━━━━━━━━━

✅ 全承認 →「all」とリプライ
❌ 個別除外 → 番号指定（例:「3,5除外」）
✏️ 修正指示 → 番号+指示（例:「2: タイトル変更して」）

GitHub PR: {PR URL}
```

### Step 3: 承認待ち

ユーザーの Slack リプライを待つ。

### Step 4: 承認処理

ユーザーの指示に基づいて処理:

**「all」の場合**:
1. 全記事の `status` を `"published"` に変更
2. PR を更新してコミット

**番号指定除外の場合**:
1. 指定された記事を PR から除外（ファイル削除）
2. 残りの記事の `status` を `"published"` に変更
3. PR を更新してコミット

**修正指示の場合**:
1. 指定された記事を修正
2. 修正後の内容を Slack で再提示
3. 承認を待つ

### Step 5: PR マージ & デプロイ

1. 承認された PR をマージ
2. GitHub Actions（または Cloudflare Pages の自動デプロイ）がビルド・デプロイを実行
3. デプロイ完了を確認

### Step 6: 完了通知

Slack に完了通知を投稿:

```
✅ *デイリーニュース公開完了（YYYY/MM/DD）*

公開記事: {N}件
{各記事のタイトルとURL}

除外記事: {M}件（あれば）
```

## 品質ティアによるレビュー負荷調整

| ティア | 条件 | Slack表示 |
|--------|------|----------|
| 🟢 自動承認候補 | スコア90+ | 「✅推奨」マーク付き |
| 🟡 標準レビュー | スコア70-89 | 通常表示 |
| 🔴 要精査 | スコア70未満 | 「⚠️要確認」マーク付き |

## ユーザー承認ポイント

このワークフロー内での承認ポイントは **1回のみ**（Step 3）。
承認を得ずに PR マージ・公開を行ってはならない。

## 注意事項

- PR のマージは必ずユーザー承認後に行う
- 修正指示があった場合は、修正内容を再度提示して承認を得る
- 全記事がリジェクトされた場合は、その日のPRをクローズする
